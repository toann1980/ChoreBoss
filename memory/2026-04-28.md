## Lesson: SQLAlchemy Autoincrement

## Session 3 — LLM Routing Solved via OpenClaw Gateway

**Toan's question:** "Can we use the OpenClaw model caller for testing instead of funding OpenAI? We did this with MemoryGraph."

**Answer: YES.** OpenClaw Gateway exposes `/v1/chat/completions` (OpenAI-compatible) on `https://10.0.0.81:18789`. Already enabled in `~/.openclaw/openclaw.json` under `gateway.http.endpoints.chatCompletions.enabled: true`. Gateway proxies to whatever model the agent is configured to use (currently GitHub Copilot → Claude/GPT). Flat-rate subscription = effectively free per-call.

**Authentication:** Bearer token from `gateway.auth.token` (already in config: `aa6b920b...`).

**Quirks found:**
1. **Model field must be `openclaw` or `openclaw/<agentId>`** — not `gpt-4o-mini` etc. Gateway routes through agent, not arbitrary model selection.
2. **~14k prompt-token bootstrap overhead per call** — gateway agent's full system prompt + memory bleeds into each request. Slow but functional.
3. **Persona bleed risk** — gateway agent's identity colored my first test response ("Bootstrap truncation acknowledged"). **Mitigation:** strong "IGNORE OTHER IDENTITY" preamble in system prompt told the model to pretend it was only the investor character.
4. **JSON sometimes wrapped in prose** — added regex fallback `\{.*\}` extract before parsing.
5. **Self-signed TLS on private LAN** — `httpx.AsyncClient(verify=False)` is acceptable in this context.

**Implementation: provider abstraction in `ai_persona.py`**
```python
LLM_PROVIDER=openclaw   # default → gateway path
LLM_PROVIDER=openai     # → direct OpenAI API (when funded)
```
Same wire format both ways. Just swap the env var.

**E2E smoke test PASSED** with `anxious_retiree` bio:
- Stayed in character ("I don't want to be burned again")
- Picked dividend-safe tickers (JNJ, KO)
- Avoided JPM despite news (loss aversion dial worked)
- Mood='cautiously optimistic', narrative coherent
- Valid JSON, no persona bleed

**Lesson:** When the user says "we did this with X before" → search the X repo first. Saved a chunk of time vs. designing from scratch.

**Lesson:** Always test the actual gateway endpoint before assuming it works the way docs describe. Discovered the `model="openclaw"` constraint by getting a clean error message on first call. Cheap learning.

### Lessons learned this session

1. **Always check if a registry/init function is actually being called at startup.** The entire MarketAux + Finnhub + RSS pipeline was built but silently dead because `initialize_default_sources()` was never called in `lifespan()`. Could have shipped months ago.

2. **feedparser.parse() is synchronous and blocks the event loop.** Any RSS fetching in async code needs `asyncio.get_event_loop().run_in_executor(None, feedparser.parse, url)` + timeout. Without this, the RSS source silently hangs and blocks all subsequent async tasks.

3. **N+1 dedup query is a silent killer.** 350 individual SELECTs for one ticker's article batch is slow enough to make scraping appear hung. Always batch with `WHERE url IN (...)`.

4. **Alembic can develop split-brain** if migrations are applied from multiple heads (e.g., multiple revisions all claiming `0009_scenario_job` as down_revision but with different revision IDs). Fix: manually update the `alembic_version` table to the single correct head.

5. **OpenClaw Gateway doesn't support `response_format: {"type": "json_object"}`** — it returns empty content. Remove that parameter and use a JSON fence fallback (`re.search(r'\{.*\}', content, re.DOTALL)`) when using the Gateway.

6. **When editing large files, the `class ClassName(Base):` line can be silently eaten.** After any model edit, do a quick `grep class.*Base db/models.py` to verify all ORM classes are intact.
